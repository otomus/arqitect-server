# Bridge (Dashboard)

The bridge is a WebSocket server that connects the [Arqitect Dashboard](https://otomus.github.io/arqitect-dashboard/) to the brain. It translates between WebSocket messages from browser clients and Redis pub/sub channels used internally by the brain.

Every dashboard session gets its own session ID (`dash_xxxxxxxxxxxx`) and authenticated identity via JWT.

## Protocol

WebSocket (ws/wss). The bridge binds to `0.0.0.0` on the configured port and upgrades HTTP connections to persistent WebSocket connections. SSL is supported when certificates are configured.

## Configuration

```yaml
ports:
  bridge: 3000        # WebSocket port

secrets:
  jwt_secret: ''      # auto-generated by the wizard

ssl:
  cert: ''            # path to TLS cert (optional)
```

| Key | Default | Description |
|---|---|---|
| `ports.bridge` | `3000` | Port the WebSocket server listens on |
| `secrets.jwt_secret` | auto-generated | HMAC secret for signing JWTs |
| `ssl.cert` | empty | Path to SSL certificate. When set, the bridge runs as `wss://` |

## Authentication

The bridge uses JWT tokens for identity.

1. Client connects via WebSocket and receives a session ID (`dash_` prefix + 12 hex chars).
2. Client sends an `auth` message with a JWT token.
3. Bridge decodes the token, verifies the user exists in cold memory, and responds with `auth_ok` (including user ID, name, and role) or `auth_error`.
4. All subsequent `task` messages from that client carry the resolved identity (`usr_{sub}`).

If the client has no token (first-time user), the bridge activates its onboarding flow, which runs entirely within the bridge without reaching the brain.

### Token refresh

The bridge implements sliding token refresh. On each `task` message, it checks whether the JWT is approaching expiry. If so, it generates a fresh token and piggybacks it on the next `brain:response` payload sent to that client.

## Channels

The bridge subscribes to a set of Redis pub/sub channels and forwards messages to connected WebSocket clients. Channels fall into two categories.

### Routed channels (per-client)

These messages contain a `chat_id` matching a specific session. Only the owning client receives them.

| Channel | Content |
|---|---|
| `brain:response` | Brain's reply to a user task |
| `brain:thought` | Internal reasoning trace |
| `brain:action` | Action the brain is about to execute |
| `nerve:result` | Result from a nerve execution |

### Broadcast channels (all clients)

Every connected client receives these.

| Channel | Content |
|---|---|
| `system:status` | CPU, memory, uptime stats (sent every 2s) |
| `nerve:qualification` | Nerve quality assessment results |
| `memory:episode` | Episodic memory consolidation events |
| `memory:tool_learned` | New tool learned notification |
| `sense:sight:frame` | Vision frame data |
| `sense:stt:result` | Speech-to-text transcription result |

## Client message types

The bridge accepts these message types from WebSocket clients:

| Type | Payload | Effect |
|---|---|---|
| `auth` | `{ token }` | Authenticate the connection |
| `task` | `{ task }` | Send a user message to the brain via `brain:task` |
| `nerve_details` | `{ name }` | Request details for a specific nerve |
| `peek` | `{ source }` | Trigger a vision capture |
| `voice` | `{ audio_b64 }` | Send audio for speech-to-text |
| `image` | `{ image_b64, prompt }` | Send an image for vision analysis |
| `sense_config` | `{ sense, key, value }` | Update a sense configuration value |
| `kill` | — | Shut down the system |

## Data flow

```
Dashboard (browser)
  │
  ▼  WebSocket
Bridge server (:3000)
  │
  ▼  Redis pub/sub
brain:task ──► Brain ──► brain:response
                           │
                           ▼  Redis pub/sub
                      Bridge server
                           │
                           ▼  WebSocket (routed by session ID)
                      Dashboard (browser)
```

## Connecting the dashboard

1. Start arqitect (`make start`). The bridge logs: `[BRIDGE] Starting Python WebSocket bridge on port 3000`.
2. Open the [Dashboard](https://otomus.github.io/arqitect-dashboard/).
3. In Settings, enter your server address (`ws://your-host:3000` or `wss://` with SSL).
4. The dashboard authenticates and starts receiving real-time updates.
